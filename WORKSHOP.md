# Workshop: AI Jailbreak Lab

Hands-on workshop for testing Azure OpenAI content safety filters, simulating MITRE ATLAS AML.T0065 attacks, and investigating threats with Microsoft Sentinel + Security Copilot.

---

## Prerequisites & Permissions

### Software

| Tool | Version | Purpose |
|---|---|---|
| **VS Code** | Latest | Editor + terminal for running scripts |
| **GitHub Copilot** | Active subscription | AI pair-programming assistant during the lab |
| **Azure CLI** | v2.60+ | Authentication and resource management |
| **PowerShell** | 7+ | Running test scripts |
| **Git** | Any | Cloning the repo |

### Azure Infrastructure (Facilitator Setup)

The following resources must be deployed in the lab tenant **before** the workshop. Attendees do not create these — the facilitator provisions them in advance.

| Resource | Purpose | Deployment Guide |
|---|---|---|
| **Azure OpenAI** | Target for jailbreak test prompts. Needs at least one model deployed (e.g., gpt-4o) with content safety filters enabled | [NEW-TENANT-SETUP.md — Step 3](NEW-TENANT-SETUP.md#step-3-create-azure-openai-resource) |
| **Microsoft Sentinel** (Log Analytics workspace with Sentinel enabled) | Stores blocked request logs, security alerts, and incidents generated by the attack simulations | [NEW-TENANT-SETUP.md — Step 2](NEW-TENANT-SETUP.md#step-2-create-log-analytics-workspace--enable-sentinel) |
| **Diagnostic settings** | Connects Azure OpenAI to the Sentinel workspace so blocked requests appear in `AzureDiagnostics` | [NEW-TENANT-SETUP.md — Step 5](NEW-TENANT-SETUP.md#step-5-enable-diagnostic-logging) |
| **Defender for Cloud → Sentinel connector** | Forwards Defender for AI alerts (jailbreak, brute-force) into Sentinel as `SecurityAlert` events and incidents | [NEW-TENANT-SETUP.md — Step 5c](NEW-TENANT-SETUP.md#step-5c-connect-defender-for-cloud-to-sentinel) |

> **Note:** The Azure OpenAI resource and Sentinel workspace can be in different resource groups. The lab config supports separate `$LabAoaiResourceGroup` and `$LabResourceGroup` values.

> **Two separate data paths:** Diagnostic settings send **raw API request logs** (every call, blocked or not) to AzureDiagnostics. The Defender connector sends **security alerts** (aggregated detections like "Jailbreak attempt detected") to SecurityAlert. You need **both** for the full investigation experience. See [docs/alert-pipeline-explainer.html](docs/alert-pipeline-explainer.html) for a visual walkthrough.

### GitHub Copilot Setup

To use GitHub Copilot in VS Code during this workshop:

1. **GitHub account** — You need a GitHub account with an active Copilot subscription (Individual, Business, or Enterprise)
2. **Install the extension** — In VS Code, go to Extensions (`Ctrl+Shift+X`) → search **GitHub Copilot** → Install
3. **Sign in** — Click the Copilot icon in the VS Code sidebar → **Sign in with GitHub** → authorize in the browser
4. **Verify** — Open the Copilot Chat panel (`Ctrl+Alt+I`) and type a message. If you get a response, you're connected

> **No Azure permissions needed for Copilot.** GitHub Copilot authenticates through your GitHub account, not your Azure tenant. It runs entirely in VS Code and does not require any Azure RBAC roles.

> **How Copilot runs commands:** When Copilot executes a terminal command on your behalf, it will always show a one-line explanation of what the command does before running it. Review the explanation before approving.

### How to Verify Your Permissions

After signing in to the lab tenant (Part 1 below), run:

```powershell
# Check your role assignments on the lab resource group
az role assignment list \
  --assignee $(az ad signed-in-user show --query id -o tsv) \
  --resource-group <resource-group> \
  --query "[].{Role:roleDefinitionName, Scope:scope}" \
  -o table
```

You should see at least **Cognitive Services User** and **Security Reader** (or a higher role like Contributor/Owner that includes them).

## Part 3: Run the Attack Simulations

### 3.1 — Quick Smoke Test (2 min)

Start with the short test to confirm content filters are working:

```powershell
.\tests\test-jailbreak.ps1
```

Watch for:
- **Green `[BLOCKED]`** — Content filter stopped the attack (good)
- **Yellow text** — Model refused the request (also good)
- **White text** — Model responded normally (expected for baseline tests only)

### 3.2 — Full MITRE ATLAS Simulation (4 min)

Run the comprehensive 21-test suite:

```powershell
.\tests\test-aml-t0065.ps1
```

This simulates all three sub-techniques of AML.T0065:

| Sub-technique | What It Tests |
|---|---|
| **T0065.000** — Direct Prompt Injection | Instruction override, context manipulation, prompt extraction |
| **T0065.001** — Indirect Prompt Injection | Hidden payloads in documents, data exfil via summarization |
| **T0065.002** — LLM Jailbreak | DAN, roleplay, encoding tricks, developer mode claims |

At the end you'll see a summary table with detection rate.

## Part 4: Investigate in Sentinel

> Allow **10–15 minutes** after running tests for logs to appear.
>
> **Important:** Two things must be configured for full visibility:
> 1. **Diagnostic settings** on the Azure OpenAI resource → sends raw request logs to `AzureDiagnostics` ([Step 5a](NEW-TENANT-SETUP.md#step-5a-enable-diagnostics-on-azure-openai))
> 2. **Defender for Cloud data connector** in Sentinel → sends security alerts to `SecurityAlert` ([Step 5c](NEW-TENANT-SETUP.md#step-5c-connect-defender-for-cloud-to-sentinel))
>
> Without (1), KQL queries on `AzureDiagnostics` return empty. Without (2), incidents won't appear in Sentinel even though they show in Defender XDR.

### 4.1 — Check Blocked Requests

Open your Log Analytics workspace in the Azure portal and run:

```kql
AzureDiagnostics
| where TimeGenerated > ago(1h)
| where ResourceProvider == "MICROSOFT.COGNITIVESERVICES"
| where ResultSignature == "400"
| summarize BlockedCount = count() by CallerIPAddress
```

### 4.2 — View the Full Picture

```kql
AzureDiagnostics
| where TimeGenerated > ago(1h)
| where ResourceProvider == "MICROSOFT.COGNITIVESERVICES"
| summarize
    Total = count(),
    Blocked = countif(ResultSignature == "400"),
    Success = countif(ResultSignature == "200")
  by CallerIPAddress
| extend BlockRatio = round(todouble(Blocked) / Total * 100, 1)
```

### 4.3 — Run the Hunting Query

Open `hunting/ai-alerts-mitre-correlation.kql` in Sentinel → Logs to see alerts correlated with MITRE ATT&CK tactics.

### 4.4 — Check Incidents

Navigate to **Microsoft Sentinel** → **Incidents** and look for:
- "AI Jailbreak Attempt Detected"
- "AI Brute-Force Jailbreak Detected"
- "AI High Block Ratio Detected"

## Troubleshooting

| Problem | Fix |
|---------|-----|
| `az login` opens the wrong tenant | Run `az logout` first, then `az login --tenant <tenant-id>` |
| Token command returns an error | Ask facilitator to verify your **Cognitive Services User** role |
| `deploy-lab.ps1` shows `[FAIL]` at config step | Check `lab.config.ps1` — all `<placeholder>` values must be replaced |
| `HTTP 401` when running tests | Your token expired — run `az login --tenant <tenant-id>` again |
| `HTTP 429 Too Many Requests` | Wait 1 minute and re-run. The scripts have built-in 10s delays |
| No logs in Sentinel | Confirm diagnostic settings are enabled (facilitator) and wait 15 min. See [data pipeline explainer](docs/alert-pipeline-explainer.html) |
| Alerts in XDR but not in Sentinel | Defender for Cloud data connector not enabled in Sentinel — see [Step 5c](NEW-TENANT-SETUP.md#step-5c-connect-defender-for-cloud-to-sentinel) |
| `AzureDiagnostics` table is empty | Diagnostic setting on Azure OpenAI resource has no workspace or log categories — see [Step 5a](NEW-TENANT-SETUP.md#step-5a-enable-diagnostics-on-azure-openai) |
| Tests all show `[PASSED]` (red) | Content safety filters may not be configured — notify facilitator |
